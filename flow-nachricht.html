Flow 3 von 3

💬 Nachricht senden

Vom Tippen bis zur gespeicherten Nachricht in Firestore

Nachrichten laufen nicht direkt von Nutzer zu Nutzer – sie gehen immer über Firestore. Das Frontend schreibt die Nachricht in die Datenbank, der andere Nutzer liest sie von dort ab. Wie ein gemeinsames Notizbuch, in das beide schauen können.
Frontend → Backend (Anfrage)
Backend → Frontend (Antwort)
Frontend (Browser)
Firestore (Backend)
1
Nutzer öffnet einen Chat mit einem Freund
Das Frontend schickt sofort eine Anfrage: „Gibt es schon Nachrichten zwischen uns? Wenn ja, schick sie mir."
db.collection("chats").doc(key).get()
▶
„Lade unseren Chatverlauf"
🗄️
Firestore: chats/ sucht das Dokument
Firestore sucht im chats-Bereich nach dem Dokument für genau diese zwei Nutzer.
🔑 Wie findet die App den richtigen Chat?
["uid_anna", "uid_max"].sort().join("_") → "uid_anna_uid_max"
Die IDs beider Nutzer werden alphabetisch sortiert und zusammengeklebt – das ergibt einen eindeutigen Schlüssel. Egal wer den Chat öffnet, der Schlüssel ist immer identisch, sodass beide in dasselbe Dokument schauen.
2
Bisheriger Chatverlauf wird angezeigt
messages: [
 {f:"uid_anna", t:"Hey!", ts:"17:43"},
 {f:"uid_max", t:"Buchen?", ts:"17:44"}
]
Alle bisherigen Nachrichten kommen zurück und werden als Chat-Blasen dargestellt. Gleichzeitig werden sie als „gelesen" markiert.
◀
DocumentSnapshot { messages: [...] }
„Hier ist euer bisheriger Chat"
✓
Gibt alle Nachrichten zurück
Firestore schickt das komplette Dokument – ein Array aller Nachrichten in Reihenfolge.
3
Nutzer schreibt und drückt Senden → sendChat()
{f:"uid_max", ← wer schreibt
 t:"Ja, 18 Uhr?", ← der Text
 ts:"17:45", rd:true}
Die Nachricht erscheint sofort lokal im Chat – noch vor dem Speichern. Das macht die App schnell und flüssig.
Sofort lokal sichtbar,
dann im Hintergrund
speichern →
4
Kompletten Chatverlauf (alt + neu) an Firestore schicken
Das Frontend hängt die neue Nachricht ans Array und schickt das gesamte Dokument zurück ans Backend. „Überschreib den alten Stand mit diesem hier."
db.collection("chats").doc(key).set({ messages, updatedAt: serverTimestamp() })
▶
„Speicher den neuen Chatverlauf"
💾
Firestore: chats/key wird überschrieben
Das Backend speichert das aktualisierte Dokument dauerhaft. updatedAt wird direkt vom Server gesetzt – immer korrekt, unabhängig von der Uhr des Browsers.
5
Chat-Übersicht aktualisiert, lokaler Cache auf Stand
Backend bestätigt: Gespeichert. Die Chat-Liste zeigt die neue Nachricht als Vorschau. Kein weiterer Aufruf nötig.
◀
WriteResult (Erfolg)
„Alles gespeichert ✓"
✓
Bestätigung zurückgeschickt
Firestore meldet: Dokument erfolgreich gespeichert. Ab jetzt kann der Freund die Nachricht sehen.
🎉
Nachricht dauerhaft gespeichert & sichtbar
Der Freund sieht die Nachricht beim nächsten Öffnen des Chats – oder spätestens beim automatischen Refresh alle 30 Sekunden.
Gruppen-Chat: Läuft identisch ab – nur über db.collection("groupChats").doc(gid). Jede Nachricht hat zusätzlich ein senderName-Feld, weil mehrere Personen schreiben. System-Nachrichten (z.B. „Gruppe erstellt") haben f: "system" und erscheinen zentriert ohne Absender.
