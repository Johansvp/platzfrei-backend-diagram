Flow 3 von 3

💬 Nachricht senden

Datenfluss vom Tippen bis zur gespeicherten Nachricht in Firestore

📨
Worum geht es hier?
Nachrichten in PlatzFrei werden nicht direkt von Nutzer zu Nutzer geschickt – sie laufen immer über Firestore. Das Frontend schreibt die Nachricht in die Datenbank, und der andere Nutzer liest sie von dort ab. Wie ein gemeinsames Notizbuch, in das beide schauen können.

Grüne Pfeile (→) = Frontend schickt Daten ans Backend  |  Gelbe Pfeile (←) = Backend antwortet
Frontend (Browser)
Firestore (Backend)
1
Nutzer öffnet einen Chat
Der Nutzer tippt auf den Namen eines Freundes. Das Frontend schickt sofort eine Anfrage ans Backend: „Gibt es schon Nachrichten zwischen uns? Wenn ja, gib sie mir."
db.collection("chats").doc(key).get()
▶
„Lade unseren Chatverlauf"
🗄️
Firestore: chats/ sucht die Konversation
Das Backend sucht im chats-Bereich nach dem Dokument für genau diese zwei Nutzer und schickt alles zurück was es findet.
🔑 Wie findet die App den richtigen Chat?
Jeder Chat braucht einen eindeutigen Namen in der Datenbank. Dafür werden die IDs beider Nutzer alphabetisch sortiert und zusammengesetzt:

["uid_anna", "uid_max"].sort().join("_") → "uid_anna_uid_max"

Egal ob Anna den Chat öffnet oder Max – der Schlüssel ist immer gleich. So landen alle Nachrichten im selben Dokument und niemand sucht im falschen Ordner.
2
Chatverlauf wird angezeigt
messages: [
  { f: "uid_anna", t: "Hey!", ts: "17:43" },
  { f: "uid_max", t: "Buchen?", ts: "17:44" }
]
Das Backend antwortet mit allen bisherigen Nachrichten. Das Frontend rendert sie als Chat-Blasen. Nachrichten werden gleichzeitig als „gelesen" markiert.
◀
DocumentSnapshot { messages: [...] }
„Hier ist euer bisheriger Chat"
✓
Gibt alle Nachrichten zurück
Firestore schickt das komplette Dokument zurück – ein Array aller Nachrichten in chronologischer Reihenfolge.
3
Nutzer schreibt und drückt Senden
{
  f: "uid_max",  ← wer schreibt
  t: "Ja, 18 Uhr?",  ← der Text
  ts: "17:45",  ← Uhrzeit
  rd: true  ← von mir gelesen
}
Die Nachricht erscheint sofort im Chat – noch bevor sie gespeichert ist. Das macht die App schnell und reaktiv. Im Hintergrund läuft dann die Speicherung.
Sofort lokal sichtbar –
dann im Hintergrund
speichern →
4
Frontend schickt den kompletten Chat ans Backend
Die App hängt die neue Nachricht an das bestehende Array an und schickt das komplette Dokument (alt + neu) an Firestore. Sinngemäß: „Überschreib den alten Stand mit diesem hier."
db.collection("chats").doc(key).set({ messages, updatedAt })
▶
„Speicher den neuen
Chatverlauf"
💾
Firestore: chats/key wird überschrieben
Das Backend empfängt das aktualisierte Dokument und speichert es dauerhaft. Der Zeitstempel „updatedAt" wird dabei direkt vom Server gesetzt – nicht vom Browser, damit er immer korrekt ist.
5
Backend bestätigt – Chat-Übersicht aktualisiert
Das Backend bestätigt: „Gespeichert." Das Frontend aktualisiert die Chat-Übersicht, damit die neue Nachricht auch als Vorschau sichtbar ist.
◀
WriteResult (Erfolg)
„Alles gespeichert ✓"
✓
Bestätigung zurück
Firestore meldet: Dokument erfolgreich gespeichert. Ab jetzt kann der Freund die Nachricht sehen.
🎉
Nachricht ist für beide Seiten sichtbar
Der Freund sieht die Nachricht beim nächsten Öffnen des Chats – oder wenn die App im Hintergrund alle 30 Sekunden automatisch die neuen Daten holt.
Gruppen-Chat läuft genauso ab – nur mit einem anderen Bereich in Firestore: groupChats/ statt chats/. Der Unterschied: Im Gruppen-Chat wird immer auch der Absendername mitgespeichert, weil mehrere Leute schreiben. Bei Direktnachrichten reicht die uid.
